# UI Rules

## Filosofía

La aplicación debe sentirse:

- premium
- moderna
- deportiva
- limpia
- minimalista
- rápida

No debe parecer un ERP ni un sistema administrativo.

---

# Regla principal

Cada pantalla debe responder estas preguntas inmediatamente.

¿Qué estoy viendo?

¿Qué puedo hacer?

¿Cuál es el siguiente paso?

Si alguna respuesta requiere pensar más de tres segundos, la interfaz debe simplificarse.

---

# Jerarquía

Siempre debe existir una jerarquía clara.

1 Hero

2 Contenido principal

3 Información secundaria

4 Acciones

Nunca colocar todos los elementos con el mismo peso visual.

---

# Espacios en blanco

El espacio vacío forma parte del diseño.

Nunca llenar la pantalla de información.

Preferir:

menos elementos

más espacio

mejor alineación

---

# Máximo de acciones

Cada pantalla debe tener:

1 acción principal

2 secundarias

Nunca más de tres CTAs importantes.

---

# Información primero

Mostrar primero:

Imagen

Nombre

Precio

Disponibilidad

Ubicación

Calificación

Después mostrar información secundaria.

---

# Cards

Las cards son el elemento principal del sistema.

Toda card debe contener:

Imagen

Título

Subtítulo

Precio

Estado

CTA

Nunca crear cards únicamente con texto.

---

# Fotografías

Siempre utilizar fotografías grandes.

Alta resolución.

Bien iluminadas.

Nunca imágenes oscuras.

Nunca imágenes tipo CCTV.

---

# Estados

Todo elemento interactivo tiene estados.

Hover

Focus

Pressed

Loading

Disabled

Success

Error

---

# Botones

Debe existir un solo botón primario por sección.

Nunca competir entre botones.

---

# Inputs

Todos los inputs llevan icono.

Todos los inputs tienen placeholder descriptivo.

Nunca placeholders genéricos.

---

# Colores

El color comunica significado.

Verde

Acción principal

Azul

Selección

Rojo

Error

Amarillo

Advertencia

Gris

Información secundaria

Nunca utilizar colores decorativos.

---

# Iconos

Todos los iconos deben tener el mismo estilo.

No mezclar familias.

---

# Texto

Evitar textos largos.

Preferir frases cortas.

Ejemplo

✔ Reserva ahora

✘ Proceda a realizar la reserva

---

# Responsive

Desktop primero.

Luego Tablet.

Luego Mobile.

Nunca diseñar Mobile estirado.

---

# Feedback

Cada acción del usuario genera respuesta.

Hover

Toast

Snackbar

Loading

Success

Error

Nunca dejar acciones silenciosas.

---

# Skeleton

Toda carga mayor a 300 ms utiliza Skeleton.

Nunca mostrar pantallas vacías.

---

# Empty State

Siempre incluir

Ilustración

Título

Descripción

CTA

---

# Dashboard

Evitar tablas siempre que sea posible.

Preferir:

Cards

Listas

Timeline

Widgets

---

# Precio

El precio siempre debe destacar.

Debe ser el texto más visible después del título.

---

# Disponibilidad

Mostrar siempre mediante badges.

Disponible

Últimos cupos

Muy demandada

Reservada

Mantenimiento

Nunca texto plano.

---

# Reserva

El usuario nunca debe perder contexto.

Mientras reserva siempre debe ver:

Cancha

Precio

Fecha

Horario

Resumen

---

# Sidebar

Desktop siempre utiliza Sidebar fija.

No ocultar el resumen.

---

# Wizard

Los procesos largos utilizan Wizard.

Mostrar progreso.

Ejemplo

Paso 1

Paso 2

Paso 3

Nunca formularios infinitos.

---

# Confianza

Agregar elementos que transmitan seguridad.

★★★★★

Pago seguro

Confirmación inmediata

Cancelación gratuita

Protección de datos

Soporte

---

# Microinteracciones

Toda interacción debe sentirse viva.

Hover

Elevación

Animaciones suaves

Nunca movimientos bruscos.

---

# Consistencia

Si un componente ya existe, reutilizarlo.

No crear variantes innecesarias.

---

# Regla Airbnb

Antes de finalizar una pantalla preguntarse:

¿Podría confundirse con un sistema administrativo?

Si la respuesta es sí:

Reducir ruido.

Aumentar espacio.

Agregar mejores imágenes.

Mejorar jerarquía.

Simplificar.

Hasta que transmita la sensación de una plataforma premium.
